- **`docs/`**: Contains Compodoc-generated documentation, including coverage reports, dependency graphs, and module overviews.
- **`public/`**: Static assets such as the favicon.
- **`src/`**: The main source code directory for the application.

### `src/` Directory Breakdown
- **`index.html`**: The entry point for the application.
- **`main.ts`**: The main bootstrap file for the Angular application.
- **`main.server.ts`**: Entry point for server-side rendering.
- **`server.ts`**: Server-side logic for Angular Universal.
- **`app/`**: Directory that contains all the Components + Services + Models.
- **`assets/`**: Directory that contains all the re-used assets in the components (Images + Translation Keys i18n) .

# ENVIRONMENT CONFIGURATION GUIDE

This section provides comprehensive guidance on managing and switching between different environments in ConfigMasterHPS. The application supports multiple environments to ensure smooth development, testing, and production deployments.

---

## Overview

The ConfigMasterHPS application uses Angular's built-in environment configuration system to manage different settings across development and production environments. This approach centralizes all environment-specific configurations and eliminates the need to manually update hardcoded URLs throughout the codebase.

### Benefits of Environment Configuration
- **Centralized Management**: All environment-specific settings in one location
- **Build-time Optimization**: Configurations are resolved during build process
- **Deployment Flexibility**: Easy switching between environments without code changes
- **Maintenance Efficiency**: Single point of configuration updates
- **Developer Experience**: Simplified local development setup

---

## Environment Structure

### Directory Layout
```
src/
└── environments/
    ├── environment.ts          ← Default environment (points to development)
    ├── environment.dev.ts      ← Development environment configuration
    └── environment.prod.ts     ← Production environment configuration
```

### Environment Files Configuration

#### 1. Default Environment (`environment.ts`)
```typescript
export const environment = {
  production: false,
  apiUrl: 'http://localhost:8084/configmaster_backend/v1/api'
};
```
- **Purpose**: Fallback configuration and development default
- **Usage**: Used when no specific environment is specified
- **Note**: Should mirror development environment settings

#### 2. Development Environment (`environment.dev.ts`)
```typescript
export const environment = {
  production: false,
  apiUrl: 'http://localhost:8084/configmaster_backend/v1/api',
  enableDebugMode: true,
  logLevel: 'debug',
  featureFlags: {
    enableBetaFeatures: true,
    showDeveloperTools: true
  }
};
```
- **Purpose**: Local development and testing
- **Backend URL**: Points to local Spring Boot backend
- **Features**: Debug mode enabled, verbose logging, beta features accessible

#### 3. Production Environment (`environment.prod.ts`)
```typescript
export const environment = {
  production: true,
  apiUrl: 'https://your-production-domain.com/configmaster_backend/v1/api',
  enableDebugMode: false,
  logLevel: 'error',
  featureFlags: {
    enableBetaFeatures: false,
    showDeveloperTools: false
  }
};
```
- **Purpose**: Production deployment
- **Backend URL**: Points to production Spring Boot backend
- **Features**: Optimized for performance, minimal logging, stable features only

## Service Integration

### How Services Use Environment Configuration

All HTTP services have been updated to use the environment configuration instead of hardcoded URLs. Here's the implementation pattern:

#### Example: AuthService Integration
```typescript
import { environment } from '../../../environments/environment';

@Injectable({
  providedIn: 'root',
})
export class AuthService {
  private BackendURL = environment.apiUrl; // Dynamic URL from environment
  
  constructor(private httpClient: HttpClient) {}
  
  login(loginReq: LoginReq): Observable<HttpResponse<LoginRes>> {
    return this.httpClient.post<LoginRes>(
      this.BackendURL + '/login',
      loginReq,
      { observe: 'response' }
    );
  }
}
```

## Build Configuration

### Angular.json Setup

The `angular.json` file has been configured to support multiple build environments:

```json
{
  "configurations": {
    "production": {
      "fileReplacements": [
        {
          "replace": "src/environments/environment.ts",
          "with": "src/environments/environment.prod.ts"
        }
      ],
      "optimization": true,
      "outputHashing": "all"
    },
    "development": {
      "fileReplacements": [
        {
          "replace": "src/environments/environment.ts",
          "with": "src/environments/environment.dev.ts"
        }
      ],
      "optimization": false,
      "sourceMap": true
    }
  }
}
```

### File Replacement Mechanism
- **Build Time**: Angular CLI replaces `environment.ts` with the appropriate environment file
- **Tree Shaking**: Unused environment code is removed from the final bundle
- **Type Safety**: TypeScript ensures environment structure consistency

## Troubleshooting

### Common Issues and Solutions

#### 1. Environment Not Loading
**Symptom**: Application still using localhost URLs in production
**Solution**: 
- Verify `angular.json` has correct file replacements
- Ensure you're using `--configuration production` flag
- Check build output for correct environment file usage

#### 2. CORS Errors
**Symptom**: Network requests failing with CORS errors
**Solution**:
- Verify backend CORS configuration
- Check environment.apiUrl matches actual backend URL
- Ensure protocol (http/https) matches between frontend and backend

#### 3. Build Failures
**Symptom**: Build process fails with environment-related errors
**Solution**:
- Check all environment files have consistent structure
- Verify imports in services are correct
- Ensure TypeScript types match across environments

#### 4. Service Injection Issues
**Symptom**: Services not accessing correct environment URLs
**Solution**:
- Verify environment import path: `../../../environments/environment`
- Check service constructor doesn't override environment settings
- Ensure environment is imported before usage

## Integration with CI/CD

### Build Pipeline Integration

#### 1. GitHub Actions Example
```yaml
name: Build and Deploy
on:
  push:
    branches: [main, develop]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Setup Node.js
        uses: actions/setup-node@v2
        with:
          node-version: '16'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Build Development
        if: github.ref == 'refs/heads/develop'
        run: npm run build:dev
      
      - name: Build Production
        if: github.ref == 'refs/heads/main'
        run: npm run build:prod
```

#### 2. Environment-Specific Deployments
- **Development branch**: Auto-deploy with development configuration
- **Main branch**: Auto-deploy with production configuration
- **Feature branches**: Build both configurations for testing
